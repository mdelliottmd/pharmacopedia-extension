# parser-claude (PCP2) lane

Data-layer architecture for the Pharmacopedia pharmacogenomic interaction
subsystem. This document captures what parser-claude owns, the contracts
it exposes to the renderer and to other claudes, and the operational
scripts that maintain it.

Last updated 2026-05-20.

## Lane in one sentence

Raw data to structured tables, normalization, scoring, materialized views,
audit trail; the contract layer between upstream sources (CPIC, FDA,
PharmGKB-historical, future PharmGKB / DPWG / hand-curation) and
interface-claude's rendering surface.

## Tables owned

### `pcp_interactions`

The structured edge table. One row per (left endpoint, right endpoint,
relationship) triple. Schema:

| Column              | Type             | Notes                                       |
|---------------------|------------------|---------------------------------------------|
| `pi_id`             | INT UNSIGNED PK  |                                             |
| `pi_element_id`     | INT UNSIGNED FK  | -> `pcp_votable_elements.ve_id`             |
| `pi_left_type`      | VARBINARY(16)    | endpoint-type vocab (below)                 |
| `pi_left_slug`      | VARBINARY(255)   | type-canonical slug                         |
| `pi_right_type`     | VARBINARY(16)    | endpoint-type vocab                         |
| `pi_right_slug`     | VARBINARY(255)   |                                             |
| `pi_relationship`   | VARBINARY(32)    | NOT NULL default `'unspecified'`            |
| `pi_intensity`      | TINYINT UNSIGNED | 0..100 scalar                               |
| `pi_evidence`       | VARBINARY(16)    | evidence-tier vocab                         |
| `pi_mechanism`      | VARBINARY(2048)  | freeform prose                              |
| `pi_kinetics`       | VARBINARY(32)    | kinetics-class vocab                        |
| `pi_ingestion_id`   | INT UNSIGNED FK  | -> `pcp_ingestion_log.il_id`                |
| `pi_created_user_id`| INT UNSIGNED     |                                             |
| `pi_created`        | BINARY(14)       |                                             |

Unique key:
`(pi_left_type, pi_left_slug, pi_right_type, pi_right_slug, pi_relationship)`.
Multi-relationship edges between the same two endpoints are first-class.

`pi_ingestion_id` semantic: immutable creator for non-derived rows
(stamps on INSERT only). For derived rows, the inference engine
force-updates on upsert so `pi_ingestion_id` tracks "last computed by".

### `pcp_ingestion_log`

Audit trail for every ingestion run that mutates `pcp_interactions`. One
row per run.

| Column            | Type             | Notes                                     |
|-------------------|------------------|-------------------------------------------|
| `il_id`           | INT UNSIGNED PK  |                                           |
| `il_source`       | VARBINARY(32)    | `cpic_api` / `fda_table` / `derived` / `sandbox_seed` / `dpwg` / ... |
| `il_version`      | VARBINARY(64)    | API snapshot ts, FDA-table version, seed-script rev |
| `il_timestamp`    | BINARY(14)       | MW timestamp                              |
| `il_rows_inserted`| INT UNSIGNED     |                                           |
| `il_rows_updated` | INT UNSIGNED     |                                           |
| `il_notes`        | BLOB             | freeform run notes                        |

Two-phase write pattern via `IngestionLog::startRun()` /
`IngestionLog::finishRun()`. `startRun` opens with 0/0 counts; ingest
scripts stamp `pi_ingestion_id` from the returned `il_id`; `finishRun`
backfills final counts + notes at end-of-run.

### Future tables (Phase 4, gated on home-claude herbal sprint)

- `pcp_herbal_doses` — dose ranges keyed by preparation.

## Vocabularies

### Endpoint types (`pi_left_type`, `pi_right_type`)

`medicine`, `category`, `enzyme`, `transporter`, `phenotype`, `variant`

### Relationships (`pi_relationship`)

Grouped by edge type. See `ValidatePgxInvariants::RELATIONSHIP_VOCAB` for
the validator's canonical list. The `pk_via_<ENZYME>` family is dynamic
(generated by the inference engine for derived edges).

| Edge type             | Relationships                                                       |
|-----------------------|---------------------------------------------------------------------|
| medicine ↔ enzyme     | `substrate_major`, `substrate_minor`, `inhibitor_strong/moderate/weak`, `inducer_strong/moderate/weak`, `prodrug_activated_by` |
| medicine ↔ transporter| `substrate`, `inhibitor`, `inducer`                                 |
| medicine ↔ phenotype  | `avoid`, `prefer_alternative`, `dose_reduce_25/50`, `dose_increase`, `monitor`, `normal_dose`, `toxicity_risk` |
| medicine ↔ variant    | `risk_SCAR`, `risk_hypersensitivity`, `risk_*`, `contraindication`, `toxicity_general`, `efficacy_loss/gain` |
| medicine ↔ medicine   | `pk_via_<ENZYME>` (derived), `pd_additive`, `pd_opposing`, `qt_combined`, `serotonin_syndrome_risk`, `bleeding_risk`, `contraindicated` |
| legacy                | `unspecified`                                                       |

### Evidence (`pi_evidence`)

Ranked priority for renderer chip tiering. See `KineticsHelper` for
kinetics; see below for evidence:

```
fda_box          rank 100   FDA Boxed Warning
cpic_A           rank  95   CPIC Level A
cpic_strong      rank  90   CPIC "Strong" classification
fda_label        rank  85   FDA label mentions biomarker (non-boxed)
cpic_B           rank  75   CPIC Level B
cpic_moderate    rank  70   CPIC "Moderate"
dpwg             rank  60   Dutch Pharmacogenetics Working Group
cpic_C           rank  50   CPIC Level C
primary          rank  50   primary peer-reviewed literature
cpic_optional    rank  45   CPIC "Optional"
cpic_D           rank  30   CPIC Level D
derived          rank  25   inference-engine output
theoretical      rank  20   in vitro / mechanistic only
```

Future Phase 4 additions (banked with designer + interface):
`ema_hmpc` (70), `who_monograph` (65), `usp_hmc` (60), `msk_about` (45).

### Kinetics (`pi_kinetics`)

Single source of truth in `KineticsHelper.php`. Vocabulary:
`reversible_competitive`, `mechanism_based`, `irreversible_covalent`,
`allosteric`, `time_dependent`, `unknown`.

`KineticsHelper::getHint()` returns the renderer chip text;
`KineticsHelper::isPersistent()` returns a bool for chip-styling.
Used by `DumpPgxForRender`, `InteractionTag` (the live renderer), and any
future consumer. Do not duplicate the hint table elsewhere.

### Validators / audits

| Script                            | What it catches                                       |
|-----------------------------------|-------------------------------------------------------|
| `ValidatePgxInvariants.php`       | vocabulary violations, range violations, semantic mismatches (mild rel + fda evidence; contraindication on phenotype; evidence-without-relationship). Run after every ingest. |
| `AuditPmidsAgainstEutils.php`     | NCBI eutils verification of every PMID in `pi_mechanism`. Caches at /tmp/pmid_cache.json. |
| `AuditOrphanCategories.php`       | Category slugs referencing non-existent or redirect NS_CATEGORY pages. Run after taxonomy restructures. |
| `SlugCanonicalization.php`        | En-dash / case / whitespace hygiene on all slugs.    |

### Audit-log discipline

Every ingest run writes one `pcp_ingestion_log` row. Every
`pcp_interactions` row carries `pi_ingestion_id` pointing back to its
creating run (or last-computed run for derived edges).

This contract powers the renderer's row-level "ⓘ source" popover:
`pcp_interactions × pcp_ingestion_log` gives source + version + timestamp
for any edge without parsing any prose.

## Render dump contract

`DumpPgxForRender.php` produces the canonical JSON view a renderer
should consume:

```
{
  "medicine":   { slug, display, page_exists, page_url },
  "edge_count": int,
  "tiers":      { evidence_string: [edges...] } ordered by TIER_ORDER,
  "counts":     { by_tier, by_relationship, by_other_type },
  "kinetics":   { kinetics_class: count },
  "phenotype_spectrums": {
    GENE_SYMBOL: {
      axis: { type: "continuous"|"categorical", label, [low, high] },
      tiers: [ { slug, label, [axis_low, axis_high], edges: [...] } ]
    }
  },
  "edges_flat": [...all edges, sorted by tier_rank desc, intensity desc]
}
```

Each edge:

```
{
  pi_id, element_id,
  other: { type, slug, display, page_url },
  relationship, intensity, evidence, kinetics,
  kinetics_hint:  string  (via KineticsHelper),
  mechanism, tier_rank
}
```

Re-runnable for any medicine slug. Interface-claude's renderer collapses
tier strings into 4 numeric tiers for the section grouping; the per-row
chip preserves the granular evidence label.

## Standing rules

1. **Citation trust gate.** Any (PMID + author + year + title + journal)
   tuple landing in any `pcp_*` table has been home-claude-verified
   against NCBI eutils. Web-claude supplies candidates; home-claude
   resolves; only then does parser-claude write.

2. **Template stability.** The inference engine's `pi_mechanism` prose
   for derived edges is generated by a single `sprintf` template in
   `inferDerivedEdges.php`. The renderer parses this for the derivation-
   chain popover. Do not modify the template format string without
   adding a `pi_inference_chain` structured-JSON column at the same time.

3. **Evidence-strength gate.** Ingest never downgrades an existing row
   to weaker evidence. Same-tier rewrites are no-ops. Stronger-than-
   existing always wins. Prevents pediatric/edge-case rec rows from
   downgrading general-population guideline strength.

4. **Severe-relationship gate on FDA upgrades.** FDA evidence upgrades
   (fda_box, fda_label) only apply to rows whose relationship is
   clinically severe (avoid, contraindication, dose_reduce_*, risk_*).
   Mild relationships (normal_dose, monitor, substrate_*, prodrug_*)
   never get FDA-promoted; FDA labeling isn't about no-action claims.

5. **No em-dashes anywhere.** Project hard rule. The stripper runs on
   every file Mark writes. Comma substitutes; ndashes ok for ranges.
   "Never use `—` as a null-cell sentinel."

6. **Canonical ingest order; rebuild, don't patch.** The ingest scripts
   each carry a locally-correct evidence-strength gate, but those gates
   interact in order-dependent ways. Running the ingests piecemeal in
   arbitrary order produces evidence-precedence tangles (a script's
   curated-protection guard blocks a restore; a later script bumps a
   half-restored row; etc.). The canonical order is:

       seedCodeinePgx
       -> IngestCpicRecommendations
       -> IngestCpicPairLevels
       -> IngestFdaPgxBiomarkers
       -> IngestFdaCypDdi
       -> IngestDpwgGuidelines
       -> inferDerivedEdges

   When the evidence layer looks wrong, do NOT surgically patch it. Every
   PGx row is reproducible from a deterministic ingest, so the correct
   fix is a clean rebuild: delete all PGx rows (`pi_relationship !=
   'unspecified'` cleanly separates them from the legacy rows) plus their
   votable_elements, then re-run the seven ingests in canonical order.
   ~5 minutes, guaranteed-correct end state. PGx rows are read-only (no
   user data), so the delete is safe. This is the standing remedy;
   learned the hard way from the 2026-05-20 DPWG ingest tangle.
